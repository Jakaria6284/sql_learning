Exploring Your First Database

📖
Understanding the Database
Step 1: Know your tables

Tutorial: Before writing any query, a SQL expert always asks: What tables do I have? How many rows? What columns? This prevents mistakes and saves hours.

Count rows in each table
SELECT 'departments' AS table_name, COUNT(*) AS row_count FROM departments
UNION ALL SELECT 'employees', COUNT(*) FROM employees
UNION ALL SELECT 'customers', COUNT(*) FROM customers
UNION ALL SELECT 'products', COUNT(*) FROM products
UNION ALL SELECT 'orders', COUNT(*) FROM orders
UNION ALL SELECT 'order_items', COUNT(*) FROM order_items;

Why This Matters

  1. 6+ tables with different purposes (who buys what, employee structure).
  2. Row counts tell you scale: 100 rows vs 10 million changes query strategy.
  3. UNION ALL stacks multiple SELECT results vertically.
  4. Real-world: When inheriting a database, always run this first to get oriented.
📖
Inspect a Single Table
Step 2: Preview the data

Real-world scenario: Your boss says "Pull me a list of products." Where's the first place to look?

Peek at products
SELECT * FROM products LIMIT 5;

Why This Works

  1. SELECT * grabs every column so you see all detail.
  2. LIMIT 5 only shows first 5 rows (prevents loading 1 million!).
  3. Now you see: id, name, category, price, stock, etc.
Check unique categories
SELECT DISTINCT category FROM products ORDER BY category;

Why This Matters

  1. DISTINCT removes duplicates so you see each value once.
  2. Answers "what product types exist?"—critical before filtering.
  3. Use case: Building dropdown menus for reports.
Part 2
SELECT & Columns (Reading Data)
Pick the right columns and create new ones from math.
1️⃣
SELECT Specific Columns
Why SELECT * is bad practice

Tutorial: In production, never fetch all columns. It's slow, clutters reports, and breaks when someone adds a new column.

❌ BAD: Select all columns
SELECT * FROM products;

Problems

  1. Fetches data you don't need (waste of bandwidth).
  2. If table adds a new column, your app might break.
  3. Reports look messy with 20 irrelevant fields.
✅ GOOD: Select needed columns
SELECT product_name, category, unit_price FROM products;

Best Practices

  1. Only 3 columns → faster, cleaner, more maintainable.
  2. Code is self-documenting: "I need name, category, and price."
  3. Real query: Sales team needs product margins by category, not inventory details.
2️⃣
Column Aliases (AS)
Rename columns for clarity

Tutorial: AS renames a column in the output. Useful for computed values and reports.

Inventory value calculation
SELECT
  product_name,
  unit_price AS price,
  stock_qty AS quantity_in_stock,
  unit_price * stock_qty AS total_inventory_value
FROM products;

Why AS Matters

  1. Without AS price, output column shows "unit_price" (less clear).
  2. Created column: unit_price * stock_qty automatically becomes total_inventory_value.
  3. Real world: Finance sends report to CEO; they need "Total Inventory Value", not a formula.
Employee salary grades
SELECT 
  full_name, 
  salary,
  salary * 0.15 AS annual_bonus
FROM employees;

Calculation Tips

  1. salary * 0.15 calculates 15% bonus on the fly.
  2. No new table needed; the math happens in query output.
Part 3
WHERE (Filtering Rows)
Keep only the data you need.
3️⃣
Basic Filters
Find rows that match conditions

Tutorial: WHERE runs BEFORE sorting/grouping. It cuts down results first, making queries faster.

Electronics over $100
SELECT product_name, category, unit_price
FROM products
WHERE category = 'Electronics' AND unit_price > 100;

How It Works

  1. category = 'Electronics' is condition 1.
  2. unit_price > 100 is condition 2.
  3. AND means BOTH must be true (not OR).
  4. Performance: Database checks WHERE first, so this scans fewer rows than fetching all then filtering in code.
Customers from USA or Canada
SELECT full_name, country, city
FROM customers
WHERE country IN ('USA', 'Canada');

IN vs OR

  1. IN ('USA', 'Canada') is cleaner than country = 'USA' OR country = 'Canada'.
  2. Much better readability; less error-prone.
4️⃣
Range & Pattern Filters
BETWEEN and LIKE for flexible searches

Tutorial: Sometimes you need a range (BETWEEN) or fuzzy match (LIKE).

Products in price range
SELECT product_name, unit_price
FROM products
WHERE unit_price BETWEEN 50 AND 150;

BETWEEN Syntax

  1. Includes both 50 AND 150 (not exclusive).
  2. Same as: unit_price >= 50 AND unit_price <= 150 but more readable.
Find "Keyboard" products
SELECT product_name, unit_price
FROM products
WHERE product_name LIKE '%Keyboard%';

LIKE Patterns

  1. % = any characters before/after.
  2. WHERE product_name LIKE 'M%' finds names starting with M.
  3. WHERE product_name LIKE '%Pro' finds names ending with Pro.
  4. Real case: Customers search for "keyboard" but catalog has "Mechanical Keyboard RGB Pro"—LIKE finds it.
5️⃣
NULL Handling
The most common beginner mistake

WARNING: This trips up 90% of SQL beginners. Pay close attention.

❌ WRONG: Using = NULL
-- This does NOT find employees with no manager
-- It returns ZERO rows, confusing everyone
SELECT full_name, job_title
FROM employees
WHERE manager_id = NULL;

Why It Fails

  1. In SQL, NULL means "unknown" or "missing value".
  2. NULL = anything is neither true nor false; it's UNKNOWN.
  3. So WHERE manager_id = NULL matches ZERO rows.
✅ CORRECT: Using IS NULL
SELECT full_name, job_title
FROM employees
WHERE manager_id IS NULL;

The Right Way

  1. IS NULL specifically tests for missing values.
  2. This correctly finds top-level employees (CEOs, department heads).
  3. Equivalently, IS NOT NULL finds employees with a boss.
Part 4
ORDER BY & LIMIT (Sorting & Top-N)
Get results in order, show only top rows.
6️⃣
Sorting (ORDER BY)
ASC for low-to-high, DESC for high-to-low

Tutorial: By default, results have no guaranteed order. ORDER BY sorts them.

Most expensive products first
SELECT product_name, category, unit_price
FROM products
ORDER BY unit_price DESC;

DESC = Descending

  1. DESC = high to low (descending).
  2. ASC = low to high (ascending, the default).
  3. Business case: Sales manager wants top revenue products first for the dashboard.
Alphabetical product list
SELECT product_name, category
FROM products
ORDER BY product_name ASC;

Multiple Columns

  1. You can sort by 2+ columns: ORDER BY category, product_name.
  2. First sorts by category, then within each category sorts by product name.
7️⃣
Top-N Results (LIMIT)
Show only the first N results

Tutorial: LIMIT stops output at N rows. Essential for top-10 lists, dashboards, and pagination.

Top 5 most expensive items
SELECT product_name, category, unit_price
FROM products
ORDER BY unit_price DESC
LIMIT 5;

Why This is Fast

  1. Database returns 5 rows instead of all 100+ products.
  2. Network bandwidth is saved.
  3. Report page shows "Top 5" not "Top 100" (CEO wants concise view).
Skip results (OFFSET)
SELECT product_name, unit_price
FROM products
ORDER BY unit_price DESC
LIMIT 5 OFFSET 5;

Pagination Pattern

  1. LIMIT 5 OFFSET 5 skips first 5, shows next 5 (items 6-10).
  2. Use for: "Next Page" button in web apps.
  3. Page 1: OFFSET 0, Page 2: OFFSET 5, Page 3: OFFSET 10, etc.

Combining SELECT, WHERE, ORDER BY
These three are the foundation of 80% of all queries.

Complete Guided Walkthrough
Real scenario: build a customer report

Scenario: Marketing wants: "Show me USA customers sorted by last order date, top 20."

Step 1: Know Your Table

What columns do customers have?

Inspect customers table
SELECT * FROM customers LIMIT 3;

Result: we see full_name, country, city, email, etc.

Step 2: Filter

Keep only USA.

USA only
SELECT full_name, country, email
FROM customers
WHERE country = 'USA';

Step 3: Add Details

But we need to know their last order. This requires a JOIN (next section), so for now show what we have ordered by name:

USA customers, sorted
SELECT full_name, email, city

LIMIT 20;

Part 6
DISTINCT (Remove Duplicates)
Many analyses need "unique values".
8️⃣
DISTINCT Basics
Show each value once

Tutorial: DISTINCT removes duplicate rows. Critical for "How many unique...?" questions.

All customer countries (duplicates removed)
SELECT DISTINCT country
FROM customers
ORDER BY country;

Why This Helps

  1. We have 50 customers but only 5 countries.
  2. SELECT DISTINCT country shows each country once.
  3. Useful for building dropdown filters in dashboards.
How many product categories?
SELECT COUNT(DISTINCT category) AS num_categories
FROM products;

Combined with COUNT

  1. COUNT(DISTINCT category) counts unique categories.
  2. We have 100+ products but only 4 categories.
Part 7
GROUP BY & Aggregates
Summarize data: COUNT, SUM, AVG, MAX, MIN.
9️⃣
Counting & Summing
Aggregate rows into summaries

Tutorial: GROUP BY collapses rows. Useful for "revenue by category", "employees by department", etc.

Sales (order count) by product category
SELECT 
  p.category,
  COUNT(oi.order_item_id) AS items_sold,
  SUM(oi.quantity) AS total_units
FROM products p
JOIN order_items oi ON p.product_id = oi.product_id
GROUP BY p.category;

How GROUP BY Works

  1. All products are bunched into their category.
  2. COUNT counts rows per group.
  3. SUM(oi.quantity) adds quantities per group.
  4. Result: Electronics 150 units sold, Furniture 200 units, etc.
Headcount and salary by department
SELECT 
  d.dept_name,
  COUNT(e.employee_id) AS headcount,
  SUM(e.salary) AS total_payroll,
  AVG(e.salary) AS avg_salary
FROM departments d
LEFT JOIN employees e ON d.department_id = e.department_id
GROUP BY d.department_id, d.dept_name;

Multiple Aggregates

  1. COUNT = how many employees.
  2. SUM(salary) = total payroll cost.
  3. AVG(salary) = average salary per department.
  4. Real-world: CFO reviews this to understand cost center budgets.
🔟
HAVING (Filter Groups)
WHERE filters before grouping, HAVING filters after

Tutorial: HAVING filters groups. Use when condition uses aggregates like COUNT or SUM.

Categories with 20+ items sold
SELECT 
  p.category,
  COUNT(oi.order_item_id) AS items_sold
FROM products p
LEFT JOIN order_items oi ON p.product_id = oi.product_id
GROUP BY p.category
HAVING COUNT(oi.order_item_id) >= 20;

WHERE vs HAVING

  1. WHERE category = 'Electronics' filters rows BEFORE grouping.
  2. HAVING COUNT(...) >= 20 filters groups AFTER aggregation.
  3. If you use aggregate in filter, HAVING is required.
Part 8
CASE WHEN (Conditional Logic)
Add IF/THEN logic directly in SQL.
⭐
Categorize Using CASE
Transform values in output

Tutorial: CASE WHEN is like IF/THEN in other languages. Test conditions and return different values.

Salary bands
SELECT 
  full_name, 
  salary,
  CASE
    WHEN salary >= 100000 THEN 'Senior (100K+)'
    WHEN salary >= 75000 THEN 'Mid (75K-99K)'
    WHEN salary >= 50000 THEN 'Junior (50K-74K)'
    ELSE 'Entry (<50K)'
  END AS salary_band
FROM employees
ORDER BY salary DESC;

How CASE Works

  1. Tests conditions top to bottom.
  2. First TRUE condition wins (ignores rest).
  3. ELSE catches anything that didn't match.
  4. Real use: Sales report showing commission tier based on annual revenue.
Product quality classification
SELECT 
  product_name,
  unit_price,
  CASE
    WHEN unit_price < 25 THEN 'Budget'
    WHEN unit_price < 75 THEN 'Standard'
    WHEN unit_price < 200 THEN 'Premium'
    ELSE 'Luxury'
  END AS price_tier
FROM products
ORDER BY unit_price DESC;

Chaining Conditions

  1. Each tier is called a WHEN clause.
  2. Creates a new output column called price_tier.

Why JOINs Matter
Real databases have multiple tables

Tutorial: Databases split data into tables (employees go in one, departments in another) to avoid duplication. JOINs reconnect them.

Employee + Department info together
SELECT 
  e.full_name,
  e.salary,
  d.dept_name
FROM employees e
INNER JOIN departments d ON e.department_id = d.department_id;

What This Does

  1. FROM employees e = start with employee rows.
  2. INNER JOIN departments d ON ... = connect to department table.
  3. ON e.department_id = d.department_id = use this column to match.
  4. Result: each employee shows their name + salary + department name.
  5. Next step: Core SQL section covers JOINS in depth.
